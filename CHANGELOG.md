## [Unreleased]

### Adicionado

- **CLI/skill de agente próprio (`apps/cli`, `@robo-multipost/agent`)**: ferramenta de linha de comando, sem dependências (usa `fetch` nativo do Node), que envolve a API pública (`/public/v1/*`) para agentes de IA (Hermes, OpenClaw, Claude, etc.). Diferente do CLI oficial do Postiz (cloud-oriented e sem automações), este é **self-hosted-aware** (lê `MULTIPOST_API_URL`/`POSTIZ_API_URL`), respeita **chave por perfil** e inclui os **flows** (automações de comentário): `profiles:list`, `integrations:list`, `posts:create|list|delete`, `flows:create|list|get|update|status|delete`, `upload:url|file`, `analytics`, `is-connected`. Acompanha `SKILL.md` (referência para o agente) e `README.md`. A pasta `apps/cli` antes era um diretório-fantasma vazio (só `.DS_Store`, não versionado) — agora tem conteúdo real. Publicação no npm é opcional; o nome `@robo-multipost/agent` é placeholder até definir o escopo.
- **Publicação em Stories do Facebook (foto e vídeo)**: ao compor um post para uma Página do Facebook, agora há o seletor **Tipo de Postagem** com as opções "Post / Reel" (comportamento atual) e "Story". Stories de foto são publicados via `/photo_stories` e os de vídeo via `/video_stories` (upload em 3 fases, com a Meta buscando o arquivo do storage). Como Story não suporta carrossel, várias mídias são publicadas como stories separados — igual ao Instagram. Não é necessário reconectar a conta: as permissões já são as mesmas solicitadas hoje. Vídeo de Story aceita no máximo 60 segundos (validação no compositor).
- **Facebook Story como destino de repost**: o Story do Facebook agora aparece como opção de destino nas regras de repost (antes só existia "Reel"). Reposts de Story/Reel do Instagram podem ser republicados como Story do Facebook (foto ou vídeo). Requer migração aditiva do banco (`FACEBOOK_STORY` no enum `RepostDestinationFormat`, aplicada via `pnpm prisma-db-push`).
- **Documentação Swagger dos demais endpoints públicos (posts, upload, integrações, perfis, analytics, vídeo)**: todos os endpoints da tag **Public API** ganharam `@ApiOperation` (descrição), `@ApiResponse` (códigos), `@ApiParam`/`@ApiQuery` e, nos principais, `@ApiBody` com exemplos copiáveis. Destaque: `POST /public/v1/posts` traz exemplos prontos (publicar agora / agendar / rascunho) no "Example Value". DTOs `UploadDto`, `VideoDto`, `GetPostsDto` e `GetNotificationsDto` anotados com `@ApiProperty`. Nova página [`docs/api/public-api.md`](docs/api/public-api.md) com referência geral e o fluxo típico no n8n (descobrir canal → upload → criar post).
- **Documentação da API pública de automações (Swagger interativo + página de referência)**: o Swagger (`/docs`) agora mostra o schema completo do corpo das automações (todos os campos, tipos, enums, limites e descrições) e ganhou botão **Authorize** (esquema de chave de API no header `Authorization`) — dá para **testar os endpoints públicos direto pelo `/docs`**. O título passou de "Postiz Swagger file" para **"Multipost API"**. Adicionada também a página [`docs/api/public-flows-api.md`](docs/api/public-flows-api.md) com referência completa: autenticação, endpoints, todos os atributos, modos de vínculo, boas práticas, exemplos (curl/n8n) e o pré-requisito do webhook da Meta.
- **Botão "Docs" para a documentação Swagger nos cards de Chave de API**: os cards "Chave de API da Organização" e "Chave de API do Perfil" (Configurações > Integrações > aba Acesso) ganharam um botão "Docs" no canto superior direito que abre a documentação Swagger da API (`/docs` do backend) em uma nova aba, com tooltip "Documentação da API (Swagger)".
- **Automações de comentário do Instagram via API pública, MCP e SDK**: agora é possível criar e gerenciar automações de resposta a comentários/stories programaticamente — antes só dava pela interface. Quando alguém comenta uma palavra-chave (ex.: "EU QUERO") em um post, a automação responde publicamente e/ou envia o link no direct. Isso fecha o cenário de ponta a ponta: gerar a imagem da receita, publicar no Instagram e criar a automação do comentário, tudo via API ou pelo agente.
  - Novos endpoints na API pública (chave de organização ou de perfil): `POST /public/v1/flows` (criar), `GET /public/v1/flows` (listar, com filtro opcional `integrationId`), `GET /public/v1/flows/:id`, `PUT /public/v1/flows/:id` (editar), `POST /public/v1/flows/:id/status` (ativar/pausar/arquivar) e `DELETE /public/v1/flows/:id` (excluir).
  - Modo de vínculo padrão `next_publication`: a automação se conecta sozinha ao próximo post publicado no canal, permitindo encadear "gerar imagem → publicar no Instagram → criar a automação" sem o cliente precisar do ID de mídia do Instagram.
  - Ferramentas MCP `createCommentAutomationTool`, `listCommentAutomationsTool` e `setCommentAutomationStatusTool` para o agente de chat (e clientes MCP como o n8n). O agente sempre confirma a configuração com o usuário antes de criar.
  - SDK `@postiz/node`: novos métodos `createFlow`, `listFlows`, `getFlow`, `updateFlow`, `setFlowStatus` e `deleteFlow`.
- **Chaves de API por perfil**: cada perfil pode gerar sua própria chave de API escopada apenas às suas integrações. A chave da organização continua existindo com acesso irrestrito a todos os perfis. Chaves de perfil funcionam na REST API pública, no MCP e no CLI.
  - A aba `Configurações > Desenvolvedores` foi renomeada para `Integrações`. No perfil Default exibe as duas chaves (organização + perfil); perfis secundários exibem apenas a chave do próprio perfil.
  - Novo endpoint `POST /profiles/:id/api-key/rotate` (admin) para gerar ou rotacionar a chave de um perfil.
  - Novo endpoint `GET /public/v1/profiles` para listar perfis disponíveis via API pública (chave de org retorna todos; chave de perfil retorna apenas o próprio perfil).
  - Segurança: tentativa de usar `?profileId` diferente da chave de perfil retorna `403 Forbidden`.

### Corrigido

- **Container em crash-loop (backend e orchestrator não subiam)**: o Dependabot havia bumpado a imagem Docker de `node:22.20-bookworm-slim` para `node:26.1-bookworm-slim` (PR #39). O `@sentry/profiling-node` não tem binding nativo para o ABI do Node 26 (`sentry_cpu_profiler-linux-x64-glibc-147.node`), então backend e orchestrator quebravam no boot com `MODULE_NOT_FOUND` e reiniciavam em loop (o frontend subia normalmente). Além disso, o Node 26 contradiz o `engines` do projeto (`>=22.12.0 <23.0.0`) e o CI, que builda em Node 22.12 — por isso o build passava mas o runtime falhava. Revertida a imagem para `node:22.20-bookworm-slim` e adicionado `ignore` de major da imagem `node` no Dependabot para não recorrer. Requer rebuild da imagem.
- **Correções pequenas trazidas do upstream Postiz (cherry-pick seguro)**: incorporados 4 fixes que não conflitam com as customizações do fork — alt text de mídia no Mastodon, tratamento de erro adequado no provider do Discord, `redirect_uri` dinâmico nos provedores de login OAuth (Google/GitHub), e permitir `localhost` ao criar um App OAuth (útil em dev/self-hosted). Triagem completa dos 119 commits do upstream em [`docs/planning/upstream-sync-triage-2026-06.md`](docs/planning/upstream-sync-triage-2026-06.md); o restante (incl. fix de XSS, hardening de SSRF e bumps de framework) exige porte manual por tocar arquivos divergentes do fork.
- **Canais do Zernio não apareciam quando havia só o perfil padrão**: ao conectar uma conta (ex.: Instagram) no perfil **Default** do Zernio e tentar adicioná-la em Lançamentos > Adicionar Canal > Zernio, o modal mostrava "Nenhuma conta conectada encontrada neste perfil", mesmo com a conta conectada no Zernio. As contas só apareciam depois de criar um segundo perfil no Zernio. Causa: o modal buscava as contas usando `selectedProfileId` (preenchido apenas quando o usuário **clica** num perfil), mas com um único perfil a seleção é automática e só atualiza `effectiveProfileId` — então a busca de contas nunca era disparada. Agora a busca usa `effectiveProfileId`, carregando as contas do perfil único (default) sem exigir clique.
- **Adicionar um segundo Instagram nas Credenciais apagava o primeiro**: em Configurações > Credenciais > Token do Instagram, ao cadastrar um segundo token de Instagram no mesmo perfil, a conta anterior sumia e parava de funcionar (automações estilo ManyChat deixavam de responder). Causa: o backend só devolve os tokens em forma redigida (`hasToken`, nunca o token em si), então ao adicionar uma conta nova o frontend remontava a lista com as contas existentes **sem** o token — e o backend descartava qualquer entrada sem token, mantendo apenas a recém-adicionada. Agora entradas sem token cuja conta já estava registrada são preservadas integralmente (token e data de renovação originais), permitindo múltiplos Instagrams por perfil. Quem já perdeu o token antigo precisa recadastrá-lo uma vez.
- **Build quebrado nos testes (CI) por erro de tipo na API pública**: a suíte `pnpm test` (ts-jest) falhava ao compilar `public.integrations.controller.ts` com `TS7018` — a const `POST_BODY_EXAMPLES` (exemplos do Swagger para `POST /public/v1/posts`) tinha arrays vazios (`tags: []`, `image: []`) cujo tipo era inferido como `any[]` implícito sob `noImplicitAny`. O `build:backend` (transpile sem type-check) não pegava, mas o CI sim, deixando todos os PRs vermelhos. A const passou a ser tipada como `Record<string, { summary: string; value: any }>`. Sem mudança de comportamento.
- **Automações criadas via API de organização não apareciam na interface**: ao criar um flow com a chave de API da organização (sem `?profileId`), ele era salvo com `profileId` nulo (org-wide) e ficava invisível na listagem de Automações, que filtra por perfil. Agora a criação nunca salva perfil nulo:
  - Chave de organização sem `profileId` → atribui ao **perfil Default** da org.
  - Chave de organização com `?profileId` → valida que o perfil pertence à org e usa ele.
  - Chave de perfil → usa o próprio perfil (como já era).
  - `profileId` inexistente na org → `400`.
  - Migração automática no boot (`StartupMigrationService`): flows órfãos já existentes (`profileId` nulo) são atribuídos ao perfil Default da organização — idempotente, então automações criadas antes do fix passam a aparecer.
- **Posts agendados via API/MCP de organização não apareciam no calendário**: ao agendar um post com a chave de organização (sem `profileId` no contexto), ele era salvo com `profileId` nulo e ficava invisível no Calendário, que filtra pelo perfil selecionado. Agora `createPost` herda o perfil **do canal de destino** quando nenhum perfil é informado — todo canal pertence a exatamente um perfil, então o post cai no calendário correto na hora (sem depender do restart que aciona a varredura de órfãos).
- **Automações do Instagram não listavam posts/stories em contas conectadas via Instagram Login**: os seletores de "post/story específico" (Wizard e API) buscavam mídias sempre em `graph.facebook.com`, retornando lista vazia para integrações conectadas via Instagram Login (IG User Token) — caso comum em instâncias self-hosted. Agora o host e o token são resolvidos por `resolveIgRoute` (Facebook Login → `graph.facebook.com`; Instagram Login → `graph.instagram.com`), igual ao restante das automações. Quem usa Facebook Login não é afetado (mesmo comportamento de antes).

### Segurança

- **Hardening de autenticação e de credenciais armazenadas (porte do upstream)**: três correções de segurança, sem tocar em billing.
  - **Sessão não confia mais em claims do JWT**: o `AuthMiddleware` agora apenas verifica a assinatura do token e **re-resolve o usuário no banco pelo id**. Antes, campos como `id`, `isSuperAdmin` e `activated` vinham do corpo do token (válidos enquanto a assinatura batesse, mesmo que o usuário tivesse sido alterado/desativado). Agora um token de um usuário inexistente/desativado é rejeitado.
  - **Conexão de canal não vaza credenciais na resposta**: o endpoint público de conclusão de OAuth (`no.auth.integrations`) passou a remover `token`, `refreshToken` e `customInstanceDetails` do objeto retornado — esses segredos ficam só no servidor.
  - **Credenciais do Skool encriptadas em repouso**: passam a ser guardadas como JSON cifrado (AES, `fixedEncryption`) em vez de JWT assinado, com **fallback de leitura** para contas conectadas no formato antigo (sem necessidade de reconectar). Porte do commit upstream `23696d2` (a parte que removia NOWPayments/billing foi deliberadamente deixada de fora).
- **Validação de upload por magic bytes + hardening do nginx (porte do upstream)**: uploads agora são validados pelo **conteúdo real** (magic bytes via `file-type`), não pelo `Content-Type`/extensão declarados (forjáveis). O `CustomFileValidationPipe` passou a detectar o tipo real, rejeitar o que não está na allowlist (`image/jpeg|png|gif|webp|avif|bmp|tiff`, `video/mp4` — **SVG e HTML ficam de fora**, por serem vetores de XSS), reescrever o nome do arquivo com a extensão correta e foi aplicado também a `POST /media/upload-simple`. O `POST /public/v1/upload-from-url` valida o buffer baixado da mesma forma. Defesa em profundidade no nginx: `/uploads/` agora envia `X-Content-Type-Options: nosniff`, `Content-Security-Policy` restritiva (sandbox) e tipos MIME explícitos. Helper compartilhado `allowed.upload.mime.ts` (import limpo, **sem `eslint-disable`** — refatorado em relação ao upstream). Porte manual do commit upstream `3ea3022`; `file-type` fixado como dependência direta. A validação magic-byte no carregamento de URL dos storages (`loadFromUrlOrDataUrl`) foi deixada de fora de propósito (data URLs de IA são confiáveis; nginx + SSRF já cobrem os vetores de serve/fetch).
- **Proteção SSRF com DNS-pinning em buscas de URL do usuário (porte do upstream)**: endpoints/serviços que baixam uma URL fornecida pelo usuário (`/public/v1/upload-from-url`, o proxy `public.controller`, e o carregamento de mídia por URL nos storages local/Cloudflare via `loadFromUrlOrDataUrl`) agora usam um `ssrfSafeDispatcher` (undici Agent) que **fixa a resolução DNS**: cada IP resolvido passa por `isBlockedIp`, fechando a janela TOCTOU que a validação só-da-URL deixava aberta (GHSA-f7jj-p389-4w45). Adicionado o guard `isSafePublicHttpsUrl` onde faltava e `undici` fixado como dependência direta. O `/upload-from-url` migrou de `axios` para `fetch` com o dispatcher. Porte manual do commit upstream `071143d` (a detecção de tipo por magic-byte virá em PR separado).
- **Sanitização do conteúdo de posts contra XSS (porte do upstream)**: o HTML do conteúdo do post agora passa por DOMPurify (`isomorphic-dompurify`) — tanto na **entrada** (transform no DTO `PostContent.content`) quanto na **saída** (página pública de preview `/p/[id]`). Só são permitidas tags de formatação seguras (`p`, `br`, `strong`, `u`, `a`, `ul`, `li`, `h1`–`h3`, `span`) e atributos seguros; `<script>`, handlers inline (`onerror`/`onclick`) e URIs `javascript:` são removidos. Novo helper `libraries/helpers/src/utils/sanitize.post.content.ts`. Porte manual do commit upstream `7236213` (não aplicável via cherry-pick por conflitar com customizações do fork).
- **Assinatura de postback do follow-gate endurecida**: o segredo HMAC deixou de ter um fallback literal público no código (que, por ser AGPL-aberto, qualquer um podia recomputar). Agora deriva de `POSTBACK_SIGNING_SECRET` → `FACEBOOK_APP_SECRET` → `INSTAGRAM_APP_SECRET` → `ENCRYPTION_KEY` → `JWT_SECRET`, com aviso em log se nenhum estiver configurado. O HMAC do payload passou de 32 para 64 bits. Sem migração: postbacks em voo expiram naturalmente na janela de 23h.
- **Servidor MCP rejeitado por clientes estritos (n8n)**: as ferramentas `integrationList` e `generateVideoOptions` não declaravam `inputSchema`, fazendo o servidor MCP serializar um schema sem `type: "object"`. Clientes tolerantes (Claude Code, Cursor) ignoravam, mas o n8n MCP Client recusava a lista inteira de ferramentas com `Invalid input: expected "object"`. Adicionado `inputSchema: z.object({})` às duas (exige restart do backend para revalidar a lista no boot).

### Segurança

- **Automações de comentário — autorização por perfil e validação de URL**: a criação de automação (via API pública, MCP ou wizard interno) agora passa por um guard único que valida que a integração pertence à organização e ao perfil do chamador (chave de API por perfil **não** cria automação no Instagram de outro perfil → `403`), que é uma conta do Instagram **ativa** (integração desativada/removida → `412`), e que a URL do botão do direct é **https pública** (bloqueia esquemas `javascript:`/`data:`/`file:` e hosts privados/locais como `localhost`/`127.0.0.1`/`169.254.169.254`). O endpoint público usa `ValidationPipe` estrito (rejeita campos extras, mitigando mass-assignment) e tem rate limit próprio (mais apertado que o global) por disparar assinatura de webhook na Meta. Endurecido também o `matchMode` (apenas `any`/`all`/`exact`) e a obrigatoriedade de `postIds`/`storyIds` quando o vínculo é `specific`.
- **Dependabot habilitado** (`.github/dependabot.yml`): arquivo existia mas estava com `package-ecosystem: ""` (inválido — bot ignorava silenciosamente). Corrigido para cobrir npm (monorepo pnpm, raiz `/`), GitHub Actions e Docker, com schedule semanal, grupos `runtime-minor-patch`/`dev-minor-patch`/`security-fixes` e majors em PRs individuais para revisão humana obrigatória.
- **Automação de auditoria Dependabot no `security-auditor`**: o subagent agora executa `gh api .../dependabot/alerts` como passo zero de qualquer auditoria, surfaçando alertas critical/high antes de revisar o diff. Gracioso — se `gh` não estiver autenticado, emite `⏭️ unavailable` e continua.

Agora a tela edita automaticamente a persona do perfil ativo (resolvido via `useCurrentProfile()`, mesmo padrão usado pelo card de "Modelos de IA"). ### Corrigido
- **Erro de IA não abre mais o modal de "Pagamento Necessário" (billing) por engano**: quando o provedor de IA (ex.: OpenRouter) retornava erro — tipicamente `402` "sem créditos" — o AI SDK lançava um `APICallError` carregando `.statusCode` + `.message`. O filtro padrão do NestJS (`BaseExceptionFilter.isHttpError`) repassava esse status como status HTTP do app, então um `402` do provedor abria o modal global de billing redirecionando para `/billing` (desabilitado neste fork), e um `401` do provedor chegava a **deslogar o usuário**. Agora `AiTextService.callWithFallback` normaliza qualquer erro de runtime do provedor para `HttpException 412` (convenção do repo: erro de IA usa 412, nunca 402) com mensagem amigável + detalhe técnico sanitizado (tokens `Bearer`/`sk-` redigidos). Resultado: o composer agora mostra um toast claro em vez do modal de cobrança. `HttpException` já controlada (ex.: 412 "credencial não configurada" do resolver) passa direto; erro sem status numérico continua virando 500. Cobre `/ai/text/caption` e todas as gerações de texto (`generatePosts`, `separatePosts`, `extractWebsiteText`). Pitfall #13 em `libraries/nestjs-libraries/src/ai/CLAUDE.md`.
- **Feedback de erro no Agente de IA**: o chat do agente (CopilotKit streaming em `/copilot/agent`) não dava **nenhum** retorno ao usuário quando o provedor falhava — o erro só aparecia no console do navegador (`runChatCompletion`). Agora o `<CopilotChat>` renderiza uma mensagem de erro amigável inline (componente `ErrorMessage` customizado) e exibe um toast (`onError`), ambos classificando o erro cru do stream nas mesmas categorias do backend (sem créditos / autenticação / limite / genérico). Novas chaves i18n (`pt`/`en`): `ai_assistant_error_title`, `ai_assistant_generic_error`, `ai_provider_no_credits`, `ai_provider_auth_error`, `ai_provider_rate_limit`, `detail_label`. - **Agente Mastra com Gemini (Google AI Studio) via OpenRouter retornava `400 INVALID_ARGUMENT`** ao tentar agendar post. O schema da tool `integration.schedule.post` tinha `settings.value: z.any()`, que Zod traduz para JSON Schema vazio `{}`. Gemini rejeita propriedades em `required[]` sem schema definido (OpenAI/Anthropic toleram). Mudado para `z.string()` com instrução no description para o agente enviar valores complexos como JSON-stringified. - **Enriquecimento de prompt de imagem/vídeo agora funciona com modelos free do OpenRouter** (Nemotron 3 Nano Omni free, Gemma 4 26B A4B free, Mistral 7B free, etc). Antes, `AiTextService.generatePromptForPicture` e `generatePromptForVideo` usavam `generateObject` do AI SDK com schema Zod `{ prompt: string }`, que exige JSON mode / structured output do provider. Modelos free do OpenRouter não suportam isso de forma confiável — retornam texto solto, markdown wrapping ou output truncado, e o parser do AI SDK falhava com `No object generated: could not parse the response`. O erro era mascarado pelo fallback (também modelo free, também falhava) e chegava ao usuário como genérico 500. Agora ambos os helpers usam `generateText` e o `.text.trim()` do retorno vira o prompt enriquecido — schema trivial não justifica o custo de exigir JSON mode. Sem impacto em modelos pagos (OpenAI direto, Anthropic via OpenRouter, etc) — todos eles também aceitam `generateText` sem perda de qualidade. 2 specs novos em `ai-text.service.spec.ts` garantindo que generateText é chamado e que `result.text` é trimado. - **Orchestrator: `app.listen(3002)` agora resiliente a `EADDRINUSE`**. Antes, qualquer falha no bind do health check HTTP derrubava o processo inteiro, criando um loop de crash de ~30s onde os workers Temporal entravam em RUNNING (drenando alguns workflows entre os crashes) e em seguida o NestJS morria. Causa raiz observada em produção: zombie state de boot anterior (provavelmente OOM partial em servidor apertado de memória) deixando socket TCP em uso. Agora, se port estiver ocupado, o erro é logado como WARN e o orchestrator continua com os workers funcionais — health check fica indisponível mas workers seguem processando workflows. Apps/orchestrator/src/main.ts.
- **Logs: bulk update de UnmatchedComment do mesmo `media_id`**. Quando um post recebia múltiplos comentários antes de o user vincular (cada um virando um `UnmatchedComment` separado por `igCommentId` distinto), vincular UM deixava os outros como `PENDING` órfãos na aba "Pendentes" mesmo depois do `FlowMediaAlias` já existir. Idempotente (only `PENDING` é tocado). Retorna `bulkBoundCount`/`bulkIgnoredCount` na resposta para a UI eventualmente mostrar "X comentários similares também movidos".

- Geração de imagem com IA quebrava em servidores rodando Node 20+ porque `undici` (fetch nativo) não suporta `data:` URLs nativamente. O fluxo `MediaController.generateImageFromText` montava `data:image/png;base64,...` e passava para `storage.uploadSimple`, que chamava `fetch(path)` e quebrava silenciosamente com erro 500. Novo helper `loadFromUrlOrDataUrl` em `libraries/nestjs-libraries/src/upload/storage.helpers.ts` detecta data URLs e usa `Buffer.from(base64)` direto (sem fetch), e `CloudflareStorage.uploadSimple` + `LocalStorage.uploadSimple` foram migrados para o helper. O controller também passou a montar o prefix `data:` antes de chamar `uploadSimple` quando `MediaService.generateImage` retorna base64 puro (caso de `isPicturePrompt=true`).
- Geração de imagem com IA falhava com 412 mesmo a chave de imagem estando configurada quando o admin não tinha credencial de TEXT. Causa: `MediaService.generateImage` chamava `AiTextService.generatePromptForPicture` (para enriquecer o prompt com detalhes de câmera, iluminação etc.) antes de gerar a imagem, e esse passo precisava de credencial TEXT. Agora o enrichment é **best-effort**: se a chamada lançar 412, loga warning e segue com o prompt original. Erros não-412 propagam normal.
- Geração de imagem com `gpt-image-1` (e modelos `gpt-image-*` em geral) falhava com `Unknown parameter: 'response_format'` da OpenAI — a nova família já retorna `b64_json` por padrão e não aceita mais o parâmetro. O `AiImageService.generateOpenAi` agora só inclui `response_format` quando o modelo é `dall-e-*` (legacy). Acabou ficando irrelevante após a limpeza do catálogo (gpt-image-1 e DALL-E foram removidos), mas mantido em código defensivamente.
- Componente `AiImage` no composer agora trata erros HTTP corretamente (412 → toast "Configure suas chaves em Settings > AI Provider"; 429 → toast "Aguarde um instante"; outros → toast com mensagem do backend), em vez de aceitar qualquer JSON como sucesso e adicionar ao state — antes um payload de erro `{statusCode, message}` virava media inválida sem `id`/`path` e disparava warning React "Each child in a list should have a unique key prop" + thumb vazia.
- PM2 dos três processos do container (orchestrator, backend, frontend) agora têm limite preventivo de memória com proteção anti-loop: `max-memory-restart` (orchestrator 2 GB, backend e frontend 1 GB cada), `min-uptime 30s`, `max-restarts 10` e `restart-delay 5s`. Antes, sob rajada de carga (ex.: anúncio Meta Ads + Reels com automação ManyChat-style disparando webhooks IG simultâneos) o orchestrator inflava com seus 33 workers Temporal, batia OOM no host (já em pressão de swap) e o cgroup do container era punido pelo kernel matando processos arbitrários. Em uma das mortes o orchestrator subiu com o `Worker.run()` em estado quebrado (PM2 reportava `online` mas o Temporal UI mostrava "No Workers Found"), deixando workflows pendurados em `Running` por horas até intervenção manual. O novo teto faz o PM2 reiniciar limpo cada processo antes do OOM kill do kernel, e a proteção anti-loop (max 10 restarts em 30s) impede crash loop infinito caso o vazamento seja persistente. Causa-raiz definitiva (host CPX32 com 7,6 GB dividido entre 18 containers + swap em uso constante) precisa ser endereçada com upgrade de VPS ou separação de serviços.
- Reconexão do canal YouTube continuava sem gravar `refresh_token` válido no banco, exigindo revogação manual do app pelo painel da conta Google a cada hora (ou desconectava sozinho). Causa-raiz em três pontos: (1) o controller de OAuth descartava `refreshToken`/`expiresIn` retornados por `authenticate()` no fluxo `reConnect` dos providers `isBetweenSteps`, sobrescrevendo o refresh_token real do Google com `body.refresh` (ID do canal/vazio); (2) na reconexão de uma conta que já havia autorizado o app, o Google pulava silenciosamente a tela de consent e devolvia `code` sem `refresh_token` associado; (3) `myaccount.google.com/permissions` nem sempre exibe o app para revogação manual, dependendo do tipo de conta. A correção combina três camadas: nova flag `keepReconnectAuthTokens` na interface `SocialProvider` (ativa apenas no YouTube — preserva tokens do `authenticate()` no fluxo `reConnect`); novo método opcional `revokeToken()` que o controller chama programaticamente antes da reconexão (POST em `oauth2.googleapis.com/revoke`), forçando o Google a esquecer o consent; e `include_granted_scopes: false` no `generateAuthUrl` do YouTube, impedindo o Google de pular a tela de consent baseado em scopes já concedidos. Resultado: clicar em "Reconectar" no canal YouTube agora exibe a tela de consent novamente e o Google passa a emitir um `refresh_token` válido, gravando-o no banco. Facebook, Instagram, LinkedIn Page e GMB seguem o comportamento default do upstream Postiz, que funciona para eles porque dependem de Page Token permanente ou access token de longa duração.
- Repost automático no X (Twitter) deixava de publicar quando a legenda da postagem original estourava o limite "weighted" do X — caracteres como `—` (em-dash) e `…` (ellipsis) contam como 2 chars no contador real, então uma legenda com 280 chars ASCII + 1 em-dash já estoura. O fluxo agora trunca a legenda em 250 chars (margem de segurança) antes de mandar para o X. Também foram acrescentados padrões em `XProvider.handleErrors` para reconhecer respostas reais da API ("Tweet text is too long", "exceeds the maximum number of characters", `paid_partnership` não autorizado), evitando que esses casos caiam no genérico "Unknown Error". Por fim, os campos opcionais `made_with_ai` e `paid_partnership` deixaram de ser enviados quando estão `false` — antes eram enviados sempre, e algumas contas sem o feature habilitado podiam recusar a requisição. Comportamento coberto por nova spec `x.provider.spec.ts` (17 casos).

### Corrigido
- Validação de webhook Instagram (introduzida em 0.5.1) passou a falhar com `Error validating application. Cannot get application info due to a system error` em diversas situações que **não** representavam problema real do usuário, bloqueando indevidamente a criação de automações em instâncias que estavam funcionando. A correção combina três camadas: (1) **Token format** — removido `encodeURIComponent` aplicado por engano no `app access token` (`appId%7CappSecret`), que a Graph API rejeita; o pipe agora é enviado cru, alinhado a `CredentialService.configureInstagramWebhook`. Bloqueio só ocorre quando há evidência positiva de problema — credenciais ausentes, ou subscription Instagram lida com sucesso mas inativa/com campos faltando, ou listas readable sem nenhuma subscription `object='instagram'` em nenhum host. Mensagens de erro mantêm contexto diagnóstico (App ID consultado e o que cada host retornou). Adicionados testes de regressão para `%7C`, Instagram-only app em `graph.instagram.com`, fallback entre hosts, fail-open em erros e fail-closed em subscription faltando.

- Analytics do X (Twitter) quebrava com `TypeError: tweets.data.data is not iterable` para qualquer conta sem tweets na janela de 100 dias ou ao chegar na última página de paginação. A Graph API v2 omite o campo `data` da resposta quando `result_count` é zero, e o `loadAllTweets` fazia spread direto sem guardar contra `undefined`. Reautenticar o canal não resolvia. Agora o método trata resposta vazia como lista vazia e só recursiona se houver `next_token` real.
- Falhas em analytics do YouTube ficavam invisíveis por causa de um `catch` que engolia silenciosamente todos os erros e retornava lista vazia. Agora o erro real (token expirado, escopo `yt-analytics.readonly` não concedido, quota excedida etc.) é logado no backend para diagnóstico, mantendo o fallback de lista vazia para a UI não quebrar.

A mensagem era passada literal na query string `?msg=Channel Updated` e renderizada sem passar pelo `t()` na página de destino. Agora o redirect carrega a string já traduzida (`Canal Atualizado`).
- Conexão de canal YouTube/Google falhava com `Erro 400: invalid_request — Missing required parameter: client_id` mesmo após configurar Client ID e Client Secret em Settings > Credenciais.
